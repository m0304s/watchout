# 커밋 메시지 전략

## 1. 커밋 메시지 형식

### 1.1 기본 형식
```bash
"[구분] <type>: <제목>
(빈 행)
- 설명(optional)
(빈 행)
<이슈 번호> <상태> <메시지>"
```

### 1.2 예시
```bash
[BE] feat: 로고 컴포넌트 구현

- 사용자 인증 후 메인 페이지에 표시되는 로고 컴포넌트 구현
- 반응형 디자인 적용

S13P21E102-36 #done 완료
```

## 2. 구분 (Scope)

### 2.1 구분 종류
- **[AI]**: AI/ML 관련 기능
- **[BE]**: Backend/Server 관련 기능
- **[FE]**: Frontend/Client 관련 기능
- **[DB]**: Database 관련 변경
- **[INFRA]**: 인프라/배포 관련
- **[DOCS]**: 문서 관련

### 2.2 구분 사용 예시
```bash
[AI] feat: 이미지 분류 모델 구현
[BE] feat: 사용자 인증 API 구현
[FE] feat: 로그인 페이지 UI 구현
[DB] feat: 사용자 테이블 스키마 추가
[INFRA] feat: Docker 컨테이너 설정
[DOCS] feat: API 문서 업데이트
```

## 3. Type 종류

### 3.1 Type 정의
- **feat**: 새로운 기능 구현
- **fix**: 버그 수정
- **docs**: 문서 관련 사항
- **style**: 스타일 변경 (형식, 세미콜론 누락 등 기능적 변경이 없는 경우)
- **refactor**: 리팩토링
- **test**: 테스트, 테스트 리팩토링
- **build**: 빌드 관련 파일 수정
- **ci**: CI 설정 파일 수정
- **perf**: 성능 개선
- **chore**: 빌드, 패키지 관련 (업데이트 등) 혹은 그 외 자잘한 수정

### 3.2 Type 사용 예시
```bash
[FE] feat: 사용자 프로필 페이지 구현
[BE] fix: 로그인 인증 오류 수정
[DOCS] docs: README 파일 업데이트
[FE] style: 코드 포맷팅 수정
[BE] refactor: API 응답 구조 개선
[FE] test: 로그인 컴포넌트 테스트 추가
[INFRA] build: Docker 이미지 빌드 설정
[INFRA] ci: GitHub Actions 워크플로우 추가
[BE] perf: 데이터베이스 쿼리 최적화
[FE] chore: 패키지 의존성 업데이트
```

## 4. 제목 작성 규칙

### 4.1 제목 규칙
- **한글**로 작성
- **명사**로 끝내기
- **마침표 사용하지 않기**
- **50자 이내**로 작성
- **명확하고 간결하게** 작성

### 4.2 제목 예시

#### 좋은 예시
```bash
[FE] feat: 사용자 프로필 페이지 구현
[BE] fix: 로그인 인증 오류 수정
[AI] feat: 이미지 분류 모델 학습 완료
[DB] feat: 사용자 테이블 스키마 추가
```

#### 나쁜 예시
```bash
[FE] feat: 사용자 프로필 페이지 구현함.  # 마침표 사용
[BE] fix: 로그인 인증 오류 수정하기      # 동사로 끝남
[AI] feat: 이미지 분류 모델 학습        # 불완전한 문장
[DB] feat: 사용자 테이블 스키마 추가하는 기능  # 너무 길고 불필요한 단어
```

## 5. 본문 작성 규칙

### 5.1 본문 규칙
- **한글**로 작성
- **무엇과 왜**에 중점
- **어떻게**보다는 **무엇을 했는지** 설명
- **빈 행으로 구분**
- **선택사항** (간단한 변경사항은 생략 가능)

### 5.2 본문 예시

#### 좋은 예시
```bash
[BE] feat: 사용자 인증 API 구현

- JWT 토큰 기반 인증 시스템 구현
- 로그인, 회원가입, 토큰 갱신 API 추가
- 보안을 위한 비밀번호 해싱 적용

S13P21E102-15 #done 완료
```

#### 나쁜 예시
```bash
[BE] feat: 사용자 인증 API 구현

- 코드를 작성했습니다
- 함수를 만들었습니다
- 테스트를 했습니다

S13P21E102-15 #done 완료
```

## 6. 이슈 번호 및 상태

### 6.1 이슈 번호 형식
- **S13P21E102-XX**: 프로젝트 이슈 번호
- **XX**: 이슈 번호 (01, 02, 03...)

### 6.2 상태 종류
- **#done**: 완료
- **#in-progress**: 진행 중
- **#todo**: 할 일
- **#blocked**: 차단됨

### 6.3 이슈 번호 및 상태 예시
```bash
S13P21E102-15 #done 완료
S13P21E102-16 #in-progress 진행 중
S13P21E102-17 #todo 할 일
S13P21E102-18 #blocked 차단됨
```

## 8. 커밋 메시지 체크리스트

### 8.1 커밋 전 체크사항
- [ ] 구분([AI], [BE], [FE] 등)이 올바른가?
- [ ] Type이 적절한가?
- [ ] 제목이 명사로 끝나는가?
- [ ] 제목에 마침표가 없는가?
- [ ] 제목이 50자 이내인가?
- [ ] 본문이 한글로 작성되었는가?
- [ ] 본문이 "무엇과 왜"에 중점을 두었는가?
- [ ] 이슈 번호가 올바른가?
- [ ] 상태가 정확한가?

### 8.2 커밋 후 체크사항
- [ ] 커밋 메시지가 명확한가?
- [ ] 다른 개발자가 이해할 수 있는가?
- [ ] 나중에 찾기 쉬운가?
- [ ] 이슈와 연결되어 있는가?

## 9. 커밋 메시지 작성 팁

### 9.1 좋은 커밋 메시지 작성법
1. **명확한 제목**: 무엇을 했는지 한눈에 알 수 있게
2. **적절한 구분**: 어떤 영역의 변경인지 명확히
3. **정확한 Type**: 변경의 성격을 정확히 표현
4. **간결한 설명**: 불필요한 내용은 생략
5. **이슈 연결**: 관련 이슈와 명확히 연결

### 9.2 피해야 할 것들
- **모호한 표현**: "수정", "개선", "업데이트" 등
- **과도한 설명**: 너무 길고 복잡한 본문
- **영어와 한글 혼용**: 일관성 있는 언어 사용
- **이슈 번호 누락**: 추적 가능성 저하
- **상태 정보 누락**: 진행 상황 파악 어려움

## 10. 팀 협업을 위한 가이드

### 10.1 커밋 메시지 리뷰
- **코드 리뷰 시 커밋 메시지도 함께 리뷰**
- **명확하지 않은 메시지는 수정 요청**
- **일관성 있는 형식 유지**

### 10.2 커밋 히스토리 관리
- **의미있는 단위로 커밋 분리**
- **작은 단위의 커밋 선호**
- **불필요한 커밋은 squash**

### 10.3 브랜치별 커밋 전략
- **feature 브랜치**: 기능별로 명확한 커밋
- **hotfix 브랜치**: 긴급 수정사항 명시
- **release 브랜치**: 배포 관련 변경사항 명시
